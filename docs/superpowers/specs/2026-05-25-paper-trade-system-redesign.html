1. ทำไมต้องรื้อใหม่

ระบบ paper trade เดิมไม่ตรงกับ trading style ของ user เลยใช้สะท้อนระบบจริงเพื่อทำเป็น EA ในอนาคตไม่ได้

Trading style ของ user (สรุป)

Entry

  • S/R levels
  • Chart patterns: double top/bottom, triple top/bottom, wedges

Win Exit

  • ปิดเองตอน "ชน R/S ฝั่งตรงข้ามไม่ผ่าน"
  • ~400 pip / ~฿500–1,000
  • ไม่รอ TP

Loss Handling

  • ไม่ตั้ง SL
  • วางไม้แก้ที่ S/R เพิ่ม
  • Lot เท่าเดิมทุกไม้ (ไม่ Martingale)

Basket Close

  • เมื่อ net positions บวก → รวบปิด เริ่มใหม่
  • ทนการวิ่งสวน > 5,000 pip
ระบบเก่าทำผิด direction: ใช้ ATR-based SL + single trade per rule + indicator-consensus entry — และ pattern discovery mining จาก trade ทุกอันรวมกัน ไม่ใช่จาก trade ของ user

2. ภาพรวมระบบ

Real Trade (MT5) │ ├──→ [1] Mirror Paper (live) │ เปิดพร้อม user, exit ใช้ R/S+momentum │ → วัด "early exit cost" │ ├──→ [2] Auto Discovery (daily cron) │ Mining: indicator combos ตอน user ชนะ │ → spawn 2 paper variants ต่อ combo │ A. single + SL (strict) │ B. basket recovery + ฿5,000 budget │ → score-based lot sizing │ ├──→ [3] Adaptive Tuning (daily cron) │ ดู paper ที่แพ้ → เสนอ filter เพิ่ม │ → A/B test → apply │ └──→ [4] Promotion Pipeline (4 gates) Stable + Validated → "EA candidate" → broadcast signal ขึ้น dashboard + noti

3. Component 1 — Mirror Paper

Goal

ทุกครั้งที่ user เปิด real order → spawn 1 paper ที่ entry/lot/direction เหมือนกัน แต่ exit ใช้ rule ของระบบ → วัดว่า "ถ้าไม่ปิดเอง จะได้/เสียเท่าไหร่"

Trigger

trade_logger.py หลัง upsert real trade ที่ order_state=filled AND open_price IS NOT NULL AND close_price IS NULL → spawn mirror paper async

Exit Logic (ใหม่)

ทุก market tick:

  1. Take profit: ราคาแตะ R1/R2 (buy) หรือ S1/S2 (sell) AND momentum indicator แสดง weakening
  2. Momentum reversal: indicator พลิกทิศ (เช่น RSI > 70 → < 60 บน buy)
  3. Hard stop: floating loss ≥ ฿2,500 (50% ของ mirror budget)
ลบ logic _average_offset เก่าใน mirror_trader.py ทิ้ง — ไม่ใช่ rule ที่สะท้อนระบบ user

Outputs

  • paper_exit_reason: tp_pivot | momentum_flip | hard_stop
  • early_exit_cost = mirror.profit − real.profit
    • +: user ปิดเร็วเสีย opportunity
    • −: user ปิดถูกจังหวะ

4. Component 2 — Auto Discovery + Spawn

Mining Direction (รื้อ)

เก่าใหม่
Mining จาก trade ทั้งหมด หา combo winrate สูง Mining จาก basket ของ user ที่ปิดเป็น net profit โดย anchor ที่ ไม้แรก ของ basket

Step 1 — Group trades เป็น basket

User เปิดไม้แก้แล้วรวบปิดทีเดียว — ถ้านับเป็น trade เดี่ยวๆ จะเข้าใจผิด เลยต้อง group ก่อน

# trades ที่ close_time ห่างกัน ≤ BASKET_CLOSE_GAP_SEC (default 1) = basket เดียวกัน
def group_into_baskets(real_trades, gap_sec=1):
    sorted_trades = sorted(real_trades, key=lambda t: t.close_time)
    baskets, current = [], [sorted_trades[0]]
    for t in sorted_trades[1:]:
        if (t.close_time - current[-1].close_time).total_seconds() <= gap_sec:
            current.append(t)
        else:
            baskets.append(current); current = [t]
    baskets.append(current)
    return baskets

Step 2 — Mining anchor ที่ไม้แรก, weight ตาม basket size

Basket sizeความหมายMining weight
1เปิดแล้วบวกเลย — clean signal1.0
2ต้องแก้ 1 ครั้ง — penalize เล็กน้อย0.7
≥ 3abnormal (user เองยังบอกว่าผิดปกติ)ทิ้ง
winning_baskets = [b for b in group_into_baskets(real_trades)
                   if sum(t.profit for t in b) > 0]

for basket in winning_baskets:
    if len(basket) > MINING_MAX_BASKET_SIZE:   # default 2 → ทิ้ง basket ใหญ่
        continue

    weight = 1.0 if len(basket) == 1 else 0.7

    first_trade = min(basket, key=lambda t: t.open_time)
    matched_slugs = matched indicators ของไม้แรก
    for combo in combinations(matched_slugs, size=2..MINING_MAX_COMBO_SIZE):
        combo_score[combo] += weight

candidates = [combo where score >= MIN_OCCURRENCE (default 5)]
ทำไมใช้ไม้แรก: entry decision ของ pattern อยู่ที่ไม้แรก ไม้แก้คือ recovery decision (คนละเรื่อง) — pattern ที่อยากให้ EA เลียนแบบในอนาคตคือ "เปิดไม้แรกตอนไหนถึงมักจะรอด"

Spawn — 2 Variants ต่อ Combo

Variant A — Strict

  • Single trade
  • SL = ATR × 2
  • No recovery

Variant B — Basket

  • Recovery basket ที่ S/R
  • No SL
  • Virtual budget ฿5,000

ทั้งคู่ใช้ entry trigger เดียวกัน แต่ exit/risk ต่างกัน → A/B compare

ไม่จำกัดจำนวน rules ที่ spawn: ยิ่งเยอะยิ่งดี — แต่ละ rule = สถานการณ์จำลองที่ต่างกัน ระบบต้องเก็บข้อมูลหลากหลายเพื่อให้ Promotion Gate (Component 4) คัดตัวที่ดีจริงได้

Dedup เท่านั้น: ถ้า combo ซ้ำกับ rule ที่ active อยู่ → skip ไม่ให้ซ้ำ

Sizing — Score-based

ScoreLotหมายเหตุ
0–400.01จุดไม่สวย
40–700.02default ของ user
70–900.05จุดสวย
90+0.10จุดสวยมาก

Score formula:

score = 0.25 × indicator_count_normalized   # จำนวน indicator match (2→0, 5→1)
      + 0.40 × pattern_winrate              # winrate ในอดีต
      + 0.20 × indicator_strength           # avg strength (เช่น RSI distance from 50)
      + 0.15 × confluence                   # มี S/R + Fib + indicator พร้อมกันไหม
Cold start: ใช้ uniform 0.02 จนครบ 100 trades แล้วค่อย calibrate ตาม actual winrate

Basket Recovery Logic (Variant B)

1 Entry signal match → open ไม้แรก (lot ตาม score)
2 ราคาวิ่งสวน → wait
3 Recovery trigger (อันใดอันหนึ่ง):
  • Primary: ราคาแตะ S/R ถัดไปฝั่งสวน → open ไม้แก้
  • Fallback: floating loss ≥ 30% ของ virtual budget (฿1,500) AND ไม่มีไม้แก้ใหม่ใน 30 นาทีล่าสุด → force open ไม้แก้ที่ราคาปัจจุบัน
ทุกไม้แก้ใช้ lot เท่าไม้แรก
4 ทุก tick เช็ค net basket P/L:
  • ถ้า net ≥ 0 AND ชน R/S ตรงข้ามไม่ผ่าน → close all (basket win)
  • ถ้า floating loss > ฿5,000 → force close all (basket blown)
5 หลังปิด → reset, รอ entry signal ใหม่
Fallback rationale: ถ้าระบบรอแต่ S/R และราคา drift ผ่าน level โดยไม่ touch (gap, fast move, S/R ห่างเกิน) basket จะ blown โดยไม่ได้ recover เลย → fallback กัน edge case นี้ คุ้มกว่ารอ "perfect entry" แล้วโดน blown

5. Component 3 — Adaptive Tuning

Goal

ดู paper ที่แพ้ของแต่ละ pattern → หา feature ที่แยก loss vs win → เสนอ filter เพิ่ม

Algorithm

สำหรับแต่ละ active rule ที่มี ≥ 30 trades:
    หา features ที่แยก loss vs win ได้ชัด:
      - session (London/NY/Asia)
      - volatility regime (ATR percentile)
      - day of week / hour bucket
      - indicator state อื่นๆ

    ถ้า feature X มี:
      loss_rate(X=true) − loss_rate(X=false) > 0.20
    → propose filter "ห้าม trade ถ้า X=true"
    → สร้าง shadow rule (active=false) ทดลอง 30 วัน
    → ถ้า shadow winrate > original + 5% → promote shadow แทน

6. Component 4 — Promotion Pipeline (4 Gates)

Paper จะ "พร้อมทำเป็น EA" ก็ต่อเมื่อผ่าน ครบทั้ง 4 gates

Gate 1 — Sample Sufficiency

  • total_trades ≥ 50
  • Active ≥ 30 วัน
  • Cover ทั้ง 3 sessions (≥ 5 trades ต่อ session)

Gate 2 — Performance

  • winrate ≥ 0.60
  • profit_factor ≥ 1.5 (gross_win / gross_loss)
  • max_drawdown ≤ ฿2,500
  • avg R-multiple ≥ 1.5

Gate 3 — Stability

  • ≥ 7 วันติดที่ผ่าน Gate 2
  • ถ้า last-14-day winrate ตก > 15% จาก lifetime → reset stable counter

Gate 4 — Walk-forward Validation

  • แบ่ง trades 70/30 (chronological)
  • Train 70% → confirm winrate ≥ 0.60
  • Test 30% → confirm winrate ≥ 0.55 (max 10% drop allowed)
ผ่านครบ 4 gates → set status='ea_candidate' → broadcast noti → user review → สร้าง MQL5 EA

9.5 Performance — design constraints

EA ส่ง market-tick ทุก 1 วินาที + ไม่จำกัด spawn → ต้อง design ให้ scale ถึง 100+ active rules ตั้งแต่ day 1

P1 — Shared indicator compute cache (ต่อ tick)

ปัญหา: Naive loop = O(rules × indicators) → 100 rules × 3 indicators × 20ms = 6 sec/tick → budget แตก

ทุก tick → compute (slug, timeframe) ครั้งเดียว เก็บใน dict, rule check = O(1) lookup

unique_keys = {(slug, rule.timeframe)
               for rule in rules for slug in rule.indicator_slugs}

# compute เพียงครั้งเดียวต่อ tick
indicator_cache = {}
for slug, tf in unique_keys:
    indicator_cache[(slug, tf)] = compute(slug, bars_cache[tf])

for rule in rules:
    results = [indicator_cache[(s, rule.timeframe)] for s in rule.indicator_slugs]

ลด complexity: O(rules × slugs) → O(unique slug-tf combos)

P2 — Fast/slow tick path split

PathFrequencyWorkload
Fastevery tick (1s)TP/SL touch, hard_stop, basket net P/L — O(open_trades)
Slowevery 15s / on bar closeindicator compute, signal status emit, entry decision

RSI/MACD ไม่เปลี่ยนใน 1 วิ — compute ทุก 15 วิเพียงพอ

P3 — paper_signals: write only on status change

ปัญหา: ถ้าเขียนทุก tick = 100 rules × 86,400 sec = 8.6M rows/day
  • เขียนเฉพาะตอน status เปลี่ยน (idle ↔ pending ↔ near ↔ active ↔ exited)
  • Last status เก็บ in-memory dict + persist ใน paper_trader_rules.last_signal_status เพื่อ restart-safe
  • TimescaleDB hypertable + retention policy 30 วัน

P4 — Denormalize paper_trader_rule_id ลง trades

ปัจจุบัน filter open paper trades ผ่าน recovery_plan->>'paper_trader_rule_id' (JSONB) → scan ทุก row
แก้: เพิ่ม column + partial index WHERE close_time IS NULL AND is_paper = true → query O(N) → O(log N)

P5 — Pattern discovery combinatorics

combinations(2..5) of 20 slugs = 21,679 combos/trade × 50 = 1M ops
แก้: MINING_MAX_COMBO_SIZE=4 + pre-filter slugs ที่ปรากฏ ≥ 3 baskets ก่อน combine

9.6 Recovery — restart safety

ระบบต้อง survive 3 scenarios: API restart, EA disconnect, full docker compose down

R1 — API restart

State ที่ปลอดภัย (อยู่ใน DB)

  • active rules
  • open paper trades
  • baskets, virtual_balance
  • win_count, total_trades

State in-memory ที่หาย → recovery

  • _rule_cache → reload จาก DB on first tick (TTL 1 hr)
  • last_signal_status → persist ใน column
  • Recovery cooldown → derive จาก trades.open_time
def can_open_recovery(basket_trades, now, cooldown_min=30):
    if not basket_trades:
        return True
    latest = max(t.open_time for t in basket_trades)
    return (now - latest).total_seconds() / 60 >= cooldown_min
restart-safe โดย default — ไม่มี state แยกที่ต้อง sync

R2 — EA disconnect / reconnect (gap)

Detection: EA ส่ง heartbeat ที่ POST /api/ea-heartbeat ทุก 60s → เก็บ last_seen_at ใน ea_status

Policy: Replay TP/SL ตามราคา reconnect

1 Detect gap = now - last_tick_time > 60s
2 สำหรับทุก open paper trade → check TP/SL touch ที่ level เดิม (ไม่ใช่ราคา reconnect)
  • buy: bid ≥ tp → close at tp; bid ≤ sl → close at sl
  • sell: mirror
3 ถ้า level อยู่ระหว่าง gap → assume touched, exit ที่ level นั้น (realistic)
4 Log paper_exit_reason='replayed_after_gap' เพื่อ audit
ทำไมเลือก replay: force-close ตัด opportunity, ignore gap ทำ P/L เพี้ยน — replay ที่ level ที่ตั้งไว้คือ middle ground

R3 — UI: EA connection status

  • Dashboard header: 🟢 EA connected (last seen: 5s ago) หรือ 🔴 EA disconnected (3 min)
  • Paper Console banner: ⚠️ EA disconnected — paper trades paused ถ้า last_seen > 2 min
  • Frontend poll /api/ea-status ทุก 10s

R4 — Full docker compose down

ทุก container ดับ → ขึ้นมาใหม่: DB ยังอยู่ (volume persist) → state ทั้งหมดยังอยู่. API restart = R1 + EA reconnect = R2. ไม่ต้องทำอะไรเพิ่ม
